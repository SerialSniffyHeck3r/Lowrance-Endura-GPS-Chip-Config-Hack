# Lowrance Endura GNS7560 GPS 리버스 엔지니어링 (Korean)

## 개요

이 레포지토리는 **Lowrance Endura 핸드헬드 GPS** 기기를 분해·수리하고,  
내부에 숨겨져 있던 **Windows CE 환경**과, 그 안에서 동작하는 **GNS7560 GPS 칩셋 설정**을 파고들며 정리한 리버스 엔지니어링 / 실험 노트입니다.

- 터치스크린이 고장난 중고 Endura 유닛을 직접 분해 및 수리
- SD 카드 기반 “펌웨어 업데이트” 메커니즘을 이용해 **OziExplorer 및 Windows CE 쉘 실행**
- Windows CE용 시리얼 터미널로 내부 GPS 모듈에 직접 접속
- 부팅 시마다 반복적으로 출력되는 `2006 Glonav Inc. GNS7560` 배너 관찰
- Windows CE 레지스트리와 공개 헤더 파일(`GN_GPS_api.h`)을 매칭하여 **GNS7560 설정값 분석**
- 레지스트리 값을 수정해 **GPS 출력 정밀도와 업데이트 주기를 2 Hz까지 향상** (그 이상에서는 불안정)

이 문서는 원본 영어 README를 바탕으로 한 **한국어 버전 상세 설명**입니다.

## Background & Motivation

예전부터 GPS 기기를 정말 좋아했고, 당연히 지금도 마찬가지다. 지금은 아예 이것을 개발하고 있으니 그럴 만도 하다.
그냥 켜놓고 화면만 보고 있어도 재밌는 종류의 장비라고 생각하는 바 있다. 

생각해 보면, 손바닥만 한 기기 안에 작은 칩 하나가 들어있고,  
그 칩이 위성 신호를 받아서 **지금 내가 어디 있는지, 어떻게 움직이고 있는지**를 다 알려준다는 건 꽤 매력적인 일이 아니겠는가?

최근 들어 모터사이클을 타기 시작했고, 자연스럽게 이런 욕심이 생겼다.

- 속도뿐만 아니라 여러 데이터 필드를 한 번에 표시해 줄 것
- 그 데이터를 기기 안에 **로그 파일 형태로 저장**해 줄 것
- 가능하면 **GPS 갱신 속도가 더 빨랐으면** 좋을 것

Garmin 계열 기기들도 고려했지만,

- 가격대가 비싸고 (정말이다. Garmin 제품은 정말이지 너무 비싸며 돈 값을 못 한다.)
- 대부분 제품이 **1 Hz 고정 GPS 갱신 속도**를 가지고 있어서 한계가 분명했다.

그러던 중 어딘가에서 이런 이야기를 들었다.

> “Lowrance Endura는 사실 안에 Windows CE가 돌아간다”

더 이상의 정보는 전혀 필요하지 않앗다.
그 즉시 EBAY를 뒤졌다. 난 어딘가에 홀린 듯 중고 Endura 유닛 두 개를 충동적으로 구매했고,  
매우 안타깝게도 그 중 하나는 터치스크린이 작동하지 않는 상태였다.

이 레포지토리는,

- 고장난 기기를 다시 살려내고  
- 못생긴 Windows CE 기본 쉘에 진입한 다음  
- 레지스트리 깊숙한 곳에 묻혀 있던 **GNS7560 GPS 수신기 드라이버 설정을 수정**하기까지의 과정을 기록한 일종의 **연구/실험 일지**라고 보면 감사하겠다.

구입한 두 개의 Endura 중 한 개는 증상이 아주 단순하고 예측 가능했다.

- 전원은 켜지고
- LCD에 UI도 정상적으로 뜨는데
- **터치가 전혀 먹지 않는 상태**

![2](disassemble1.jpg)

기기를 분해해서 내부를 보면,

- 그 주변의 납땜/커넥터 상태가 상당히 수상해 보인다.

그리고 이 증상은 나에게 매우 익숙했다. 이전에도 이렇게 죽은 기기를 살려본 적이 있기 때문이다! 

난 감압식 터치를 별로 좋아하지는 않는다... 

![2](disassemble2.jpg)

그래서:

- 다시 부팅해 보았다.

결과적으로 터치스크린이 정상적으로 살아났다.  
이제 손에는:

- 마음껏 뜯어보면서 실험할 “수리 완료 유닛” 한 대

이렇게 두 개의 샘플이 준비된 셈이다.

## 부트 플로우 탈취: Endura 앱에서 OziExplorer로

![1](Boot_Screen.jpg)

Endura 펌웨어에는 흥미로운 동작이 하나 있다.

- 부팅 시 SD 카드가 삽입되어 있고
- 그 SD 카드에 특정 이름을 가진 실행 파일이 존재하면
- 원래 Endura 순정 앱 대신 **그 실행 파일을 실행**한다.

명목상으로는 “펌웨어 업데이트용 실행 파일”을 돌리기 위한 메커니즘이지만,  
사실상 **런처를 바꿔치기할 수 있는 진입점**이 된다.

Windows CE용 서드파티 내비게이션 소프트웨어로 **OziExplorer**라는 프로그램이 있다.  
이 실행 파일을 SD 카드에 넣고, Endura가 찾는 업데이트용 exe 이름으로 맞춰주면:

1. 부팅 시 Endura 순정 UI 대신 OziExplorer가 실행되고
2. OziExplorer가 제공하는 환경 설정/툴을 이용해
3. `explorer.exe`를 띄우는 바로 가기를 만들 수 있다.

![3](WinCE_Home.jpg)

`explorer.exe`가 실행되면  
우리가 잘 아는, 그 **촌스러운 Windows CE 기본 쉘**이 그대로 나타난다.

즉... 이 말인 즉슨

> Endura는 커스텀 셸만 씌워진 Windows CE 핸드헬드 컴퓨터에 가깝다.

라고 볼 수 있다는 결론에 도달할 수 있다.

## GPS 모듈과의 통신

![3](Serial_Port.jpg)

Windows CE 쉘에 진입한 후,  
Windows CE용 시리얼 터미널을 설치하고  
재미있어 보이는 COM 포트들을 하나씩 찍어 보기 시작했다.

그중 특정 포트에서 GPS 관련 반응이 있었고,  
결국 **COM3가 GPS 전용 포트**라는 것을 확인했다.  
(순정 Lowrance 앱도 내부적으로 이 포트를 하드코딩해 사용하는 것으로 보인다. 아님 말고..)

의심되는 GPS 포트에 연결해 보니,  
항상 이런 문장이 제일 먼저 출력되었다.

> **2006 Glonav Inc. GNS7560 All rights reserved.**

그 뒤에는 표준적인 **NMEA 포맷 GPS 문장들**이 계속해서 흘러나온다.

흥미로운 점은 다음과 같다.

- 완전 콜드 부팅 후에만 이 문장이 나오는 것이 아니라
- 시리얼 세션을 닫았다가 다시 열 때마다
- **항상 같은 부트 배너가 먼저 출력된다.**

처음에는 이것이 다소 이상하게 느껴졌다.

- 만약 이 포트가 GPS 칩의 하드웨어 UART에 그냥 바로 물려 있었다면,
- 단순히 “터미널 프로그램을 다시 열었다”는 이유만으로
  부트 메시지가 반복적으로 나오는 것은 자연스럽지 않다.

즉, **중간 어딘가에 소프트웨어 레이어가 끼어 있을 가능성이 높다는 가설을 세울 수 있다!**

## “왜 부트 메시지가 매번 반복될까?”

일부 GPS 모듈이 전원 인가 시점에만  
자신의 버전 정보나 배너를 한 번 출력하는 것은 흔한 일이다.

하지만 이 경우에는:

- 시리얼 포트를 닫았다가 다시 열 때마다
- 마치 새로 부팅한 것처럼
- `2006 Glonav Inc. GNS7560...` 메시지가 반복된다.

이런 동작은,

- 내가 보고 있는 대상이 단순한 “하드웨어 UART 라인”이 아니라
- 포트가 열릴 때마다 내부 상태를 초기화하는 **소프트웨어 UART/드라이버 레이어**일 수 있다는 점을 강하게 시사한다.

이후, 예전 Android·Windows CE 기반 내비게이션 기기를 해킹하는 글들을 찾기 위해  
포럼(특히 러시아 중고차/내비게이션 커뮤니티 등)을 뒤져본 결과:

- **Glonav GNS7560** 칩이
- 종종 **소프트웨어 시리얼 드라이버** 뒤에 붙어서 사용되며
- 그 설정은 OS 내부에서 관리된다는 이야기를 발견할 수 있었다.

그렇다면 Windows CE에서는 다음과 같은 가설을 세울 수 있다.

> 이 GPS “드라이버”의 설정은 OS 어딘가에 저장되어 있을 것이고,  
> Windows CE의 특성상 그 “어딘가”는 대부분 **레지스트리**다.

![3](Regdeit_Version.jpg)

일부 Windows CE 빌드에는 regedit가 기본 포함되어 있지 않지만,  
ARM용 Windows CE Regedit는 온라인에 그야말로 '널려 있으며' 비교적 쉽게 구할 수 있다.  
적절한 버전을 구해 실행해 보니, 예상대로 레지스트리 안에 여러 단서들이 있었다.

## Windows CE 레지스트리에서 드라이버 찾기

Windows CE에서는 많은 내장 드라이버들이

> **HKEY_LOCAL_MACHINE\Drivers\BuiltIn**

아래에 등록된다.

그래서 GPS나 Glonav와 관련이 있어 보이는 키를 찾기 위해  
레지스트리를 전체적으로 탐색했다.

결국, 기대했던 이름을 그대로 발견하게 된다.

> **HKEY_LOCAL_MACHINE\Drivers\BuiltIn\Glonav-GPS**

이 키 아래에는 여러 개의 설정 값들이 있었고,  
그 이름들이 어디서 많이 본 것 같았다.

![3](gpsset1.jpg)  
![3](gpsset2.jpg)  
![3](gpsset3.jpg)

GNS7560에 대해 인터넷을 뒤져 보면서  
사실상 유일하게 찾을 수 있었던 공개 자료가 하나 있었는데,

- `GN_GPS_api.h` 라는 헤더 파일이다.
- 이는 동일한 GNS7560 수신기를 사용하는 Android 기기의 소스 트리에서 가져온 것이다.

링크:  
<https://github.com/Nu3001/hardware_rk29_gps/blob/master/gns7560/inc/GN_GPS_api.h>

이 헤더 파일 안에 등장하는 심볼 이름들 상당수가  
레지스트리 `Glonav-GPS` 키 아래에 있는 값 이름들과 **거의 1:1로 매칭**되었다.

이로써 그림이 정리된다.

- Endura의 내부 GPS 모듈은 **Glonav GNS7560**이고
- 해당 모듈은 `Glonav-GPS`라는 이름의 **Windows CE 드라이버**를 통해 제어되며
- 런타임 설정은 전부 **레지스트리 값**으로 유지된다.

그리고 이는 곧,

> 레지스트리 값을 수정함으로써  
> GPS 동작을 직접 튜닝할 수 있다는 뜻이기도 하다.

![3](Reg_trip.jpg)

추가로, 레지스트리를 살펴보면서 흥미로운 점을 하나 더 발견했다.  
시스템 내부의 여러 상태값이 전부 **레지스트리 형태로 저장**되고 있었다.

- 시스템 버전 정보
- 시스템 업데이트 히스토리
- Trip Computer, GPS 관련 데이터
- 순정 Lowrance Endura 앱이 다루는 거의 모든 변수들

사실상 이 기기의 많은 내부 상태가 레지스트리 안에 그대로 노출되어 있는 셈이다.

## GNS7560 GPS 설정 튜닝

`Glonav-GPS` 레지스트리 키 아래에 있는 설정 값들과  
`GN_GPS_api.h` 헤더 파일 안의 필드 이름을 비교하면서  
각 설정이 무엇을 의미하는지 대략적으로 유추할 수 있었다.

이후 실제로 다음과 같은 항목들을 수정해 보았다.

- 위치/정밀도 관련 설정
- NMEA 출력 주기 (업데이트 레이트)

![4](resulrt.gif)

실험 결과:

- 기본 상태: 약 1 Hz 주기로 NMEA 출력
- 튜닝 후: **2 Hz 출력**까지는 안정적으로 동작
- 2 Hz를 넘어가도록 설정하면: 일정 시간이 지나면 모듈이 멈추거나 불안정해짐

즉, 이 하드웨어 + 펌웨어 조합에서  
**실용적인 상한선은 2 Hz 정도**로 보인다.

그럼에도 불구하고,  
기기 외부에서 펌웨어 이미지를 수정하지 않고,

- 레지스트리 편집과
- Windows CE용 시리얼 터미널만으로

실제 GPS 업데이트 속도를 끌어올렸다는 점은 나름 만족스러운 결과였다.

## OziExplorer가 NMEA 스트림을 처리하는 방식에 대한 가설

GPS 리프레시 레이트를 2 Hz로 설정했을 때,  
OziExplorer 상의 거의 모든 데이터 필드가 **2 Hz로 갱신**되기 시작했다.

- 시간, 날짜
- 기압, 온도

등, GPS와 직접적으로 관련이 없고  
다른 센서(다른 프로토콜)를 통해 측정되는 값들까지도 모두 2 Hz로 갱신되는 것을 볼 수 있었다.

이를 바탕으로, OziExplorer가 NMEA 스트림을 처리하는 방식에 대해  
아주 단순한 가설을 세울 수 있다.

- NMEA 스트림의 도착 자체가 **데이터 갱신 트리거** 역할을 한다.
  - 새로운 NMEA 문장이 올 때마다, 기기에서 가져올 수 있는 모든 데이터 필드를 한 번씩 갱신하려고 시도한다.
- NMEA 스트림이 들어오지 않는다면, 데이터는 갱신되지 않는다.
- 다만, 사용자가 앱 설정에서 의도적으로 GPS를 비활성화한 경우에는  
  GPS 외의 데이터 필드들이 **1 Hz 정도의 내부 타이머 기반**으로 갱신되는 것으로 보인다.

정리하면,

> GPS로부터 들어오는 NMEA 데이터가 “전체 데이터 갱신 루프”의 기준이 되고,  
> GPS를 끄면 별도의 타이머 기반 루프로 넘어가는 구조일 가능성이 있다.

## 이 레포지토리가 의미하는 것 

이 레포는 다음과 같은 것이 **아니다.** 오해 말도록. 

- 완성된 커스텀 펌웨어
- “그냥 설치만 하면” 모든 게 해결되는 패키지
- 상용 소프트웨어에 대한 크랙/패치 툴

대신, 이 레포는 아래 내용을 담은 **연구/실험 노트**에 가깝다.

- 터치스크린이 고장 난 Endura 유닛을 분해·수리한 과정
- SD 카드 기반 “업데이트 실행 파일” 메커니즘을 이용해:
  - 순정 Endura 앱 대신 **OziExplorer**를 실행하는 방법
  - `explorer.exe`를 띄워 Windows CE 기본 셸에 진입하는 과정
- 내부 GPS 포트에 연결해 얻은 시리얼 로그와 관찰 내용
- Windows CE 레지스트리에서 export한:
  - 원본 `Glonav-GPS` 설정 값
  - 2 Hz로 튜닝한 설정 값
- 레지스트리 키와 `GN_GPS_api.h` 안의 필드 이름 사이의 대응 관계 정리

## 디스클레이머

이 프로젝트는 **교육, 실험, 그리고 약간의 재미**를 위한 목적으로 진행되었다.

- 어차피 뭐 펌웨어 자체랑은 상관이 없긴 하다만... 펌웨어 이미지 자체는 배포하지 않는다.
- 상용 바이너리를 수정하거나 재배포하지 않는다.
- 모든 실험은 소유권이 본인에게 있는 기기에서만 수행했다.

즉 나는 이런 것을 권장하는 것이 절대 아니다!:

- 유료 지도나 온라인 서비스의 라이선스를 우회하기 위한 용도
- 본인 소유가 아닌 기기에 대한 무단 개조 및 침해

반대로, 이 레포가 장려하고 싶은 방향은 다음과 같다.

- 겉으로는 봉인되어 있는 것처럼 보이는 장비를 열어 보고, 내부가 어떻게 돌아가는지 이해해 보기
- 개인적인 목적과 호기심을 위해 기기를 개조해 보기
- 그리고 그 과정을 **재밌게 즐기는 것!**

재밌고 유익하면 장땡이다. 그렇지 않겠는가?
